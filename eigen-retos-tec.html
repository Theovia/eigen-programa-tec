Parte II

Catálogo de actividades

El stack de Eigen sigue la frontera actual de la plataforma de Cloudflare — incluyendo lo anunciado esta misma semana en Agents Week 2026 (días 1–4, del 13 al 16 de abril): Durable Object Facets, Dynamic Workers, Sandboxes GA con identity-aware auth, CLI unificado cf, Local Explorer, Cloudflare Mesh + Workers VPC, servidores MCP remotos, Workflows durables en GA, catálogo expandido de modelos (Claude 4.6, GPT-5.4, Llama, Mistral), Agents SDK next edition con Think, Agent Lee, Registrar API, Browser Run con Human-in-the-Loop, Outbound Workers para Sandboxes, Project Think con execution ladder de 5 tiers, AI Search con hybrid retrieval, AI Gateway unificado con 14+ proveedores, Artifacts (Git-compatible storage para agentes), Cloudflare Email Service en public beta, y PlanetScale Postgres/MySQL integrado vía Hyperdrive. Lo que el estudiante toca aquí es lo que este mismo trimestre está tocando un ingeniero en San Francisco o Berlín.

A · Plataforma edge & compute
  1. Servicios HTTP sobre Cloudflare Workers con Hono — middleware, routing anidado, typing estricto de extremo a extremo. Cada módulo corre en ~330 PoPs globales con tiempo de arranque en milisegundos.
  2. Uso del nuevo CLI unificado cf liberado en Agents Week (abril 2026) para despliegues, secretos, bindings y ~3,000 operaciones de API desde una sola herramienta coherente — reemplazando el patchwork de Wrangler + dashboard manual.
  3. Depuración local con Local Explorer, la herramienta recién GA para inspeccionar D1, KV, Durable Objects y Queues en ambiente de desarrollo sin tener que chequear en producción.
  4. Gestión de environments (staging, production), promociones entre ambientes, y rollbacks instantáneos con un solo comando.
  5. Uso de Workers Static Assets para servir frontends y SPAs desde el mismo Worker que sirve la API — sin CORS, sin deploys separados, sin bucket externo.
B · Persistencia y datos
  1. Modelado de esquemas en D1 (SQLite edge distribuido): migraciones versionadas con cf d1 migrations, consultas con window functions y CTEs sobre datos productivos.
  2. KV para sesiones, caches de baja latencia y feature flags, entendiendo su modelo de eventual consistency y cuándo es la herramienta equivocada.
  3. R2 para almacenamiento de archivos (cero egress fees), con URLs firmadas para descargas directas desde el cliente.
  4. Durable Objects para estado consistente por entidad — un DO por paciente, por huésped, por conversación. Strong consistency, storage co-located con cómputo.
  5. Durable Object Facets (beta liberada abril 2026): instanciar DOs con bases SQLite aisladas por entidad dinámica, permitiendo que cada app/tenant tenga su propia base sin pre-provisionamiento. Pieza clave para construir plataformas multi-tenant.
  6. Hyperdrive para conexiones aceleradas a Postgres externo (Supabase, Neon) desde Workers, sin cold starts de pool ni latencia adicional.
  7. Queues para trabajos asíncronos confiables — reintentos con backoff, dead-letter queues, consumo batch, idempotencia.
  8. Workflows durables (con mejoras GA en abril 2026) para pipelines multi-paso que sobreviven reinicios, timeouts y fallas intermitentes — el reemplazo correcto de "cron + scripts".
C · Agentes de IA y LLMs
  1. Integración de la API de Claude con tool use, prompt caching (TTL de 5 min y 1 h), memoria externa, y contextos extendidos de 1 M tokens cuando el caso lo justifica.
  2. Diseño de tools que el agente invoca: agendar en Google Calendar, cobrar vía Stripe, buscar en Supabase, hacer broadcast por WhatsApp Cloud API. Aquí está el 80% del oficio — un agente sin buenas herramientas es un bot de chat.
  3. Dynamic Workers: cargar código generado por LLM en runtime, ejecutarlo aislado, y persistir estado con Facets. Habilita la clase de productos "platform + AI" donde los usuarios generan mini-apps dentro de tu app.
  4. Cloudflare Sandboxes (lanzadas abril 2026): entornos aislados persistentes con shell, filesystem y procesos background, que inician bajo demanda y retoman exactamente donde quedaron. Útiles para agentes que ejecutan código arbitrario o scripts de usuario.
  5. Workers AI para inferencia local de modelos open-source (embeddings, reranking, generación pequeña) sin salir de la red de Cloudflare, con los nuevos soportes GA de structured outputs, tool calling nativo y contextos largos.
  6. Catálogo expandido de modelos (incluyendo GPT-5.4 agregado este mes, Claude Opus 4.6, Gemini, Llama, Mistral) — todos tras la misma interfaz de Workers AI, lo que permite comparar y rutear por costo/latencia.
  7. Vectorize para memoria semántica — búsquedas por similitud, recomendaciones, detección de anomalías y contexto persistente para agentes.
  8. AI Gateway para ruteo y observabilidad de llamadas a LLMs: caching, rate limiting, logging de prompts y atribución de costo por modelo/feature.
  9. Servidores MCP remotos (Model Context Protocol) hospedados sobre Workers para exponer herramientas a agentes externos (Claude Desktop, Cursor, Copilot) con autenticación OAuth y rate limiting por cliente.
  10. Eval suites rigurosas: conjunto de casos de prueba reales, rúbricas de evaluación, detección de regresiones cuando cambias prompt o modelo. Medir accuracy, latency, cost — la diferencia entre "ingeniería de IA" y "ingeniería de prompts".
D · Seguridad y acceso
  1. API tokens scannable (GA abril 2026): emisión, rotación y detección automática cuando un token se filtra a un repositorio público.
  2. Permisos resource-scoped (GA abril 2026): darle a cada Worker, agente o usuario únicamente los permisos mínimos que necesita. Zero-trust real, verificable en CI.
  3. OAuth apps internas con la nueva visibilidad GA — qué tokens existen, quién los ha usado, cuándo. Auditoría sin hojas de cálculo.
  4. Cloudflare Mesh + Workers VPC (lanzados este mes): dar a agentes acceso privado a bases y APIs internas sin tunneles manuales ni VPNs frágiles. El reemplazo correcto de "abramos el firewall un ratito".
  5. Gestión de secretos con cf secret y variables por environment — nunca hardcodear credenciales, nunca commitear .env.
E · Pagos, canales y backoffice
  1. Integración con Stripe para cobros únicos y subscripciones: webhooks verificados criptográficamente, idempotencia de eventos, manejo de chargebacks y reembolsos.
  2. Integración con Google Calendar vía OAuth2: flow completo (consent → access token → refresh token), diferencia entre cuentas personales y Workspace, revocación y recuperación de refresh tokens perdidos.
  3. WhatsApp Cloud API directo o vía ManyChat para mensajería masiva y conversacional: templates aprobados, mensajes con media, webhooks entrantes, rate limits de Meta.
  4. Arquitectura correcta de ManyChat con Dynamic Content (no External Request, que cachea mal) para lógica conversacional que ManyChat no soporta nativamente.
  5. Supabase como DB operacional para algunos verticales: Row Level Security para multi-tenant, triggers para auditoría, edge functions cuando tiene sentido.
F · Calidad, observabilidad y oficio
  1. Tests unitarios y de integración con Vitest + Miniflare, corriendo el Worker localmente contra mocks de D1, KV y R2, con coverage que sirve de contrato.
  2. Captura de errores con Sentry (integración nativa con Workers) y síntomas UX con PostHog — distinguiendo errores fatales, warnings de negocio y ruido ignorable.
  3. Triage de incidentes: qué es bug real, qué es flaky, qué escala al on-call, qué se responde directamente al cliente sin abrir ticket.
  4. Code review con PRs chicos (<300 líneas), commits legibles en formato convencional, checks de CI que bloquean merges si fallan.
  5. Documentación técnica viva: ADRs (Architecture Decision Records), runbooks de incidentes, postmortems honestos sin blame.
  6. Observabilidad-primero: correlation IDs, trace spans y métricas custom antes de declarar un feature "done". Si no se puede medir en producción, no está terminado.
G · Operación de clientes y proyectos
  1. Coordinación paralela de 3–5 cuentas activas (clínicas, hoteles, farmacias) con stand-ups semanales, minutas accionables y SLA de respuesta documentado y auditado.
  2. Documentación viva por cuenta en Notion: decisiones, compromisos, riesgos, próximos pasos. Una página por cliente que cualquiera puede leer en cinco minutos y entender el estado completo.
  3. Ritmo semanal con clientes: agenda preparada, preguntas escritas, hand-off al equipo técnico, follow-up por escrito en menos de 24 horas.
  4. QA funcional antes de cada release con checklists de smoke test y happy-path; automatizar lo repetitivo para poder revisar a fondo lo nuevo.
  5. Triage de Sentry y PostHog: qué convierte en ticket en Linear, qué responde el equipo directamente al cliente, qué se ignora activamente.
H · Datos, reporting y analytics
  1. SQL productivo: CTEs, window functions, time-series, agregaciones para métricas de negocio (revenue, conversion, retention, cohorts).
  2. Extracción de datos heterogéneos (Meta Ads API, PostHog, Sentry, Stripe, WhatsApp, Supabase) y normalización en un modelo semántico común.
  3. Dashboards HTML interactivos con Chart.js, filtros por dropdown, distribución como archivos self-contained que el cliente abre desde el email sin login ni VPN.
  4. Alertas de anomalía con umbrales adaptativos: caída de conversión, CPA fuera de rango, errores en picos — escaladas automáticamente al canal correcto.
  5. Automatización de reportes mensuales con Workers + Queues cron-triggered, entrega por email o WhatsApp, versionado de snapshots históricos.
  6. Catálogo de métricas versionado: cada métrica tiene dueño, fórmula y fuente escrita — acabando con "depende quién pregunte".
I · Plataforma extendida de Cloudflare
  1. Browser Rendering API — Chrome headless hospedado en la red de Cloudflare, invocable desde un Worker para capturar screenshots de dashboards ejecutivos, renderizar PDFs de reportes, hacer scraping estructurado de sitios que requieren JS, y automatizar pruebas end-to-end. Para Eigen: generar el PDF mensual de Dr. Porras directo desde el Worker, sin pagar un servicio externo de Puppeteer-as-a-service ni mantener un Chrome propio.
  2. Cloudflare Containers (beta, 2026) — workloads que no corren en el runtime V8 de Workers (Python con dependencias nativas, ffmpeg, Whisper, modelos ML custom, scripts que necesitan un POSIX real) se ejecutan en contenedores aislados con integración nativa a Workers, D1, R2 y Durable Objects. Para Eigen: transcribir audio de llamadas con Whisper, correr OCR sobre recetas médicas, transformar video de huéspedes — sin salir de la red de Cloudflare ni incurrir en egress fees de AWS.
  3. Analytics Engine — base de datos time-series integrada, accesible desde Workers con env.ANALYTICS.writeDataPoint(...); escribe métricas custom a escala masiva (millones de puntos por segundo) y se consulta con SQL. Para Eigen: medir latencia por tool-call de agente, conversiones por vertical, uso por cliente — todo sin montar Grafana externo ni pagar Datadog.
  4. R2 Data Catalog (Iceberg tables, 2026) — capa analítica sobre R2 con metadata catalog administrado por Cloudflare, consultable directamente por Snowflake, DuckDB, Spark o ClickHouse a través de JDBC/HTTP. Para Eigen: almacenar eventos históricos de todos los verticales (visitas, reservas, transacciones, sesiones de chat) como datos analíticos open-format sin mover información entre clouds ni pagar licencias.
  5. Cloudflare Pipelines (2026) — ingesta HTTP de eventos con buffering automático, batching, transformaciones inline y persistencia final a R2 o Data Catalog. Es el reemplazo correcto de Kinesis Firehose y Kafka administrado, sin ZooKeeper, sin cluster, sin operación. Para Eigen: capturar cada evento de agente, cada click en landings, cada webhook de ManyChat, en un solo pipeline declarativo que alimenta el analytics stack.
  6. Email Workers — procesar correos entrantes con un Worker en lugar de servidor IMAP propio. Recibes el mensaje crudo MIME, lo parseas, clasificas y respondes automáticamente con lógica completa. Para Eigen: la bandeja contacto@drmarcos.com atendida por un agente LLM que clasifica por intención (agendar, cobrar, urgencia), responde de forma natural y escala al humano solo cuando detecta ambigüedad o sensibilidad.
  7. Email Routing — ruteo de direcciones custom (doctor@tudominio.com → Gmail personal) sin montar MX records, sin Zoho, sin Google Workspace. Soporta reenvío selectivo por regex, forwards a Email Workers para procesamiento programable, y alias ilimitados por dominio.
  8. Turnstile — alternativa privacy-first a reCAPTCHA que no guarda cookies ni rastrea usuarios con fingerprinting. Funciona invisible en la mayoría de casos (challenge transparente), con fallback visual solo cuando detecta comportamiento sospechoso. Para Eigen: proteger formularios de leads en landings de doctores y hoteles sin degradar la tasa de conversión ni depender de Google.
  9. Cloudflare Access (Zero Trust) — SSO empresarial para aplicaciones internas sin montar Auth0 ni Okta. Configuras que admin.eigen.dev solo deja entrar a gente con email @eigen.dev, con MFA obligatoria, políticas por path, auditoría completa y sesiones con TTL configurable. Para Eigen: proteger dashboards internos de operación, admin panels de cada cliente, servicios de staging — todo sin escribir una línea de auth propia.
  10. Cloudflare Gateway (Zero Trust) — filtro DNS y HTTP para la red del equipo: bloquea phishing, malware y sitios no autorizados; inspección TLS; políticas por usuario o grupo; logs de navegación para compliance. Relevante cuando Eigen empiece a contratar gente con acceso a datos sensibles de clientes médicos o financieros.
  11. Workers Logs (GA) — logging nativo de Workers con retención de 3 a 7 días, búsqueda por correlation ID, filtros por status/endpoint/error, acceso vía dashboard y API. Complementa Sentry para los casos donde no quieres exportar cada log fuera de Cloudflare (p.ej. logs que contienen PII médico o de pago).
  12. Tail Workers — un Worker que recibe en vivo los eventos de ejecución de otros Workers (logs, traces, errores, métricas) y los procesa programáticamente. Para Eigen: pipeline de observabilidad custom que envía solo errores críticos a Slack, spans de trace a Honeycomb, métricas de latencia a Analytics Engine, y compacta el resto en R2 para auditoría.
  13. Zaraz — tag manager server-side que corre en el edge: GA4, Meta Pixel, TikTok Pixel, Hotjar, LinkedIn Insight — todos configurados una vez en Zaraz, ejecutándose desde el Worker en lugar del browser del usuario. Para Eigen: landings de cliente con tracking completo de conversión sin cargar 15 scripts de terceros que bajan PageSpeed y violan políticas de privacidad.
  14. Smart Placement — Cloudflare decide automáticamente en qué PoP ejecutar tu Worker para minimizar latencia hacia los backends que usa. Si tu Worker llama Supabase en us-east-1, Smart Placement lo coloca cerca de ahí aunque el usuario final esté en Ciudad de México. Relevante cuando el tiempo de ida-y-vuelta al backend domina la latencia total (común con DB externa).
  15. Versioned Deploys + Gradual Rollouts — cada deploy tiene versión inmutable con ID propio, puedes hacer rollouts progresivos (5% → 25% → 100%), rollback instantáneo a cualquier versión anterior, y pruebas A/B entre versiones con ruteo determinístico por usuario. Para Eigen: desplegar el cambio del agente de Dr. Marcos al 10% de pacientes, observar las evals y métricas de satisfacción, y solo pasar a 100% si no hay regresión.
  16. Cron Triggers — Workers que se ejecutan en horarios definidos sin HTTP, invocados por el scheduler nativo de Cloudflare (cron syntax estándar). Hasta 1,000 cron jobs por cuenta sin infraestructura adicional, sin EC2 con cron, sin Celery beat. Para Eigen: generar reportes mensuales, enviar recordatorios diarios, compactar logs, todo declarativo en wrangler.toml o cf.toml.
  17. Service Bindings — un Worker llama a otro Worker sin cruzar la red (cero latencia, cero costo adicional) vía bindings tipados. Sustituye el patrón "microservicios vía HTTP interno" por invocaciones directas con typesafety end-to-end. Para Eigen: el Worker de "agente" llama al Worker de "billing" y al Worker de "calendar" como si fueran funciones locales.
  18. Workers for Platforms (dispatch namespaces) — hospedar código de cliente aislado en tu propia red: cada cliente tiene su propio Worker bajo un dispatcher común, con aislamiento fuerte, límites configurables, y deploys independientes. Es la primitiva que habilita construir un SaaS donde tus propios usuarios (hoteleros, clínicas) pueden escribir lógica custom sobre tu plataforma. Potencialmente el cimiento de "Eigen Atlas como plataforma".
  19. WebSockets + Durable Objects Hibernation — WebSockets persistentes hasta millones de conexiones concurrentes con hibernación automática (no acumulas billing mientras el socket está idle, y el DO despierta cuando llega actividad). Para Eigen: features en vivo cuando aplique — dashboards que se actualizan en tiempo real, conversaciones con typing indicators, notificaciones push a cliente sin polling.
J · Resto de la plataforma de Cloudflare
  1. WAF (Web Application Firewall) — reglas managed que cubren el OWASP Top 10 + reglas custom escritas en Wirefilter (lenguaje tipo BPF), modos simulate/log/block por regla, análisis en vivo de tráfico por país/ASN/path. Para Eigen: proteger landings y APIs sin escribir middleware propio, compliance básico para clientes médicos.
  2. Bot Management — detección de bots con modelos ML entrenados sobre el tráfico agregado de Cloudflare, asignando un bot score (1-99) por request, con rate limiting adaptativo por reputación IP y distinción sofisticada entre scrapers legítimos (Google, GPTBot autorizados) y hostiles (credential stuffing, inventory hoarding).
  3. DDoS Protection avanzado — mitigación automática L3/L4/L7 sin configuración, Advanced TCP Protection contra ataques sofisticados, historial de ~200 Tbps de capacidad neta — suficiente para absorber ataques de nivel nation-state sin degradar servicio.
  4. Rate Limiting nativo — rate limiting configurable por URL, header, cookie, país o IP con ventanas slide y burst control, sin escribir una línea de Worker. Soporta rate limiting por combinación de atributos (p.ej. "100 requests por IP por hora para /login con status 401").
  5. Page Shield — detecta scripts de terceros modificados sospechosamente en el cliente (ataque Magecart, card skimming), esencial para compliance PCI DSS 4.0 en landings que reciben pagos. Alerta si un tercero inyecta código inesperado.
  6. API Shield — validación de schema OpenAPI por request, JWT validation en el edge, discovery automático de endpoints no documentados, abuse detection específico para APIs (secuencias anómalas, enumeración de IDs). Para Eigen: cada API pública de cliente pasa por schema guard.
  7. mTLS (mutual TLS) — certificados de cliente verificados en el edge antes de que el request llegue al Worker, ideal para comunicación server-to-server, APIs B2B y webhook authentication sin compartir tokens.
  8. Magic Transit — BGP advertising de tus propios prefijos IP hacia Cloudflare, protección DDoS L3/L4 transparente para infraestructura no-HTTP (servidores de juego, VoIP, IoT). El tráfico regresa al origen limpio vía GRE tunnel o direct connect.
  9. Magic WAN — SD-WAN as a service: conexiones cifradas entre oficinas, clouds y data centers usando la red de Cloudflare como backbone global privado, con políticas de ruteo y seguridad uniformes.
  10. Spectrum — proxy TCP/UDP para protocolos no-HTTP (SSH, RDP, Minecraft, MQTT, proprietary), con protección DDoS, ruteo Anycast y acceso zero-trust vía Access. Ideal cuando tienes servicios legacy que no hablan HTTP pero necesitan exponerse seguros.
  11. Argo Smart Routing — ruteo dinámico de requests vía la red privada de Cloudflare hacia tus backends, evitando peering points congestionados. Típicamente reduce latencia p50 un 30% y p99 un 40% para rutas internacionales.
  12. Load Balancing — balanceo geo-aware con health checks HTTP/TCP, weight por pool, failover automático entre regiones, session affinity por cookie/header, Monitors con alertas. Sustituye ELB/ALB/GLB para arquitecturas multi-cloud.
  13. Cache Reserve + Tiered Cache — capa de cache permanente sobre R2 (Cache Reserve) + topología jerárquica edge→regional→origin (Tiered Cache) para aumentar hit ratio global y reducir carga en origen a prácticamente cero para contenido estático pesado.
  14. Cloudflare Images — storage + transformación on-the-fly (resize, crop, format conversion, WebP/AVIF/JPEG XL), CDN integrado, signing de URLs para acceso privado. Reemplaza Cloudinary/imgproxy/sharp-server con una línea de URL.
  15. Cloudflare Stream — video hosting con transcoding automático (HLS/DASH adaptive bitrate), player embebido con DRM opcional, live streaming con sub-segundo glass-to-glass, recording, analytics de viewing por segmento.
  16. Workers Calls (WebRTC SFU) — Selective Forwarding Unit para WebRTC a escala global, disponible desde Workers — construye videoconferencia, voz grupal o livestream interactivo sin montar Jitsi ni LiveKit propio. Billing por minuto de pista, no por servidor.
  17. AutoRAG (nuevo 2026) — RAG administrado end-to-end: subes documentos a R2, Cloudflare maneja ingestión, chunking, embedding, vector storage en Vectorize, retrieval y generación con Workers AI. Para casos donde no quieres mantener tu propio pipeline RAG.
  18. D1 read replicas (2026) — réplicas read-only en múltiples regiones para escalar lectura, con eventual consistency controlada y elección explícita de "read from replica" por query, ideal para dashboards y reportes.
  19. D1 Sessions API — consistent reads dentro de una sesión lógica para garantizar que lecturas posteriores a un write ven el write, crítico para flujos transaccionales sin perder la escalabilidad edge.
  20. R2 Event Notifications — dispara Workers, Queues o Pipelines cuando un objeto se crea, modifica o elimina en R2. Patrón clave para pipelines reactivos: "cuando suban un PDF, procésalo con OCR" en una línea de configuración.
  21. Realtime / Pub-Sub (MQTT) — broker MQTT distribuido en el edge para IoT, chat, telemetría y dashboards en vivo. Hasta millones de clientes concurrentes por tenant, con retención configurable y autenticación por cert o token.
  22. Durable Object Alarms — DOs con scheduler interno: el DO programa una alarma que se dispara en tiempo definido y ejecuta lógica (reintentar, cobrar, notificar). Es el reemplazo correcto de cron para workflows por-entidad ("a los 7 días del no-pago del paciente X, mándale recordatorio").
  23. Cloudflare for SaaS — custom hostnames para tus clientes (acme.tu-saas.com → acme.com), con certificados TLS emitidos y rotados automáticamente por Cloudflare, soporte hasta 500,000 dominios por account, fallback DNS en caso de dominio primario caído. Indispensable si Eigen construye plataforma white-label.
  24. DNS autoritativo + Registrar + SSL — servidor DNS de Cloudflare (latencia p50 <10ms globalmente, DNSSEC, GeoDNS, CAA records, DDNS), registrar de dominios al costo de mayorista sin markup, certificados gestionados (Universal + Advanced con wildcards y subdominios) con rotación automática.
  25. Logpush — exportación streaming de logs (HTTP requests, Workers Trace Events, Firewall Events, AI Gateway, Zero Trust) a destinos externos: S3, GCS, Azure Blob, Datadog, Splunk, Elasticsearch, Sumo Logic. Formato JSON batcheado con schema estable.
  26. Real User Monitoring (RUM) + DEX — métricas de Core Web Vitals (LCP, FID, CLS, INP) del tráfico real de tus sitios sin instrumentar JS, más Digital Experience Monitoring para sintetizar perceived performance desde ubicaciones globales simuladas.
  27. Workers OAuth Provider — librería oficial de Cloudflare para convertir tu Worker en OAuth 2.1 provider con PKCE, dynamic client registration (RFC 7591) y device flow. Usado como backbone del sistema de MCP remoto, pero aplica a cualquier API que quieras exponer a terceros con auth robusta.
  28. Arquitectura de Isolates V8 (fundamento) — cada Worker corre en un V8 isolate independiente: no container, no VM, arranque en menos de 5 ms, aislamiento de memoria fuerte entre tenants, y el modelo que hace posible ~50,000 QPS por PoP y sobrevender CPU sin degradar latencia. Entender este fundamento cambia cómo diseñas: cold starts no existen, pero tampoco hay procesos de background sin DO.
K · Profundidad de Solutions Architect
  1. Cloudflare Tunnel (cloudflared) — conexión saliente encriptada desde infraestructura on-prem o cloud privada hacia la red de Cloudflare, sin abrir puertos inbound ni montar VPN. Es cómo expones legacy servers, databases internas, o servicios en una VPC corporativa al mundo a través del edge. Combinado con Access, permite zero-trust a cualquier servicio TCP/UDP sin refactor.
  2. Primitivas del runtime de Workers — ctx.waitUntil() para trabajo async después de responder (logs, analytics, cache warming); ctx.passThroughOnException() para fallar suave y no romper el request; subrequests internos vía fetch() a otros Workers; el ciclo de vida de handlers (fetch, scheduled, queue, email, tail, alarm) y cómo cada uno recibe context distinto. Son las primitivas que distinguen a un Solutions Architect de alguien que solo pega código.
  3. AI Gateway Guardrails — detección inline (antes de hablar al modelo, después de recibir respuesta) de PII, prompt injection, jailbreak attempts, lenguaje tóxico, y output constraint violations. Configurable por política: bloquear, redactar, alertar. Para Eigen: crítico en flujos médicos donde un paciente puede compartir sin querer su CURP o número de tarjeta en el chat.
  4. Workflows avanzado — steps atómicos con estado persistente automático entre reinicios, retries con backoff declarativo, sleep hasta timestamp específico, signals externos (workflow.signal("approved")) para human-in-the-loop, y compensación estilo saga para flujos que deben "deshacerse" si falla un paso posterior. Para Eigen: el motor correcto de flujos de pago con aprobación manual, onboarding de cliente con pasos humanos, y agendas con confirmación diferida.
  5. Cloudflare Snippets + Transform Rules + Rulesets — modificaciones de request/response sin escribir Workers: Snippets para lógica JS ligera de ~5 KB, Transform Rules para header rewriting (CORS, CSP, security headers), y Rulesets como motor unificado de reglas compositable (WAF, Rate Limiting, Cache, Config todos hablan el mismo lenguaje). Para Eigen: A/B testing a nivel header, país-gating de landings, CSP dinámica por cliente — sin tocar código.
  6. Waiting Room — cola virtual administrada en el edge cuando un sitio enfrenta demanda mayor a su capacidad (apertura de citas, flash sale, launch de temporada). Usuarios esperan con una pantalla personalizada, pasan a la app en orden controlado, y el origen nunca se satura. Para Eigen: relevante cuando Cosalá abra reservaciones de temporada alta o cuando Dr. Marcos anuncie slots nuevos a lista de 3,000 pacientes.
  7. Authenticated Origin Pulls + Origin CA — el origen solo acepta requests con un certificado de cliente que Cloudflare firma privadamente, cerrando la puerta a que alguien bypass el edge y ataque el origen directamente. Origin CA emite certificados gratis válidos solo dentro de la red de Cloudflare. Complemento de WAF + mTLS para arquitecturas serias.
  8. Matriz de decisión KV vs DO vs D1 vs R2 vs Cache API — KV para lecturas globales eventualmente consistentes a baja latencia (feature flags, config); Durable Objects para estado consistente por entidad con lecturas locales; D1 para datos relacionales con SQL; R2 para blobs; Cache API para cache HTTP programático por PoP. Un Solutions Architect es capaz de defender la elección con números (TTL, write QPS, tamaño, costo) y no por default. Esta pregunta viene en toda entrevista técnica seria.
  9. Patrones de compliance (HIPAA, PCI DSS 4.0, GDPR, LFPDPPP) — cómo combinar Cloudflare para cumplir: PII redaction en Logpush, encryption at rest en R2 con server-side KMS, data residency selectiva (US-only workers), audit trails immutables en R2 con Object Lock, Access policies con MFA y device posture para PHI, Page Shield para PCI DSS 4.0 sobre landings de pago. Extremadamente relevante porque Eigen opera con pacientes (datos médicos, LFPDPPP en México) y pagos (PCI).

Parte III

Arquitecturas de referencia

Un Solutions Architect no memoriza productos: compone patrones. Estas son las diez arquitecturas que cubren el 90% de lo que Eigen construye sobre Cloudflare. Cada una resuelve un problema recurrente y combina productos específicos con una razón — y con trade-offs que vale la pena saber antes de implementar.

01

Agente conversacional multicanal

Clientes B2C (pacientes, huéspedes) esperan atención 24/7 conversacional en WhatsApp con acceso real a agenda, pagos y escalamiento a humano. Un bot de flujos rígidos falla; un LLM sin herramientas ni memoria también.

Componentes
WhatsApp Cloud API Worker ingestor Durable Object por conversación Claude API · tool use · cache 1h Vectorize (memoria) Tools: Google Calendar · Stripe · Supabase Workflows (transacciones) Analytics Engine
Flujo
Usuario → WhatsApp → Worker (verifica firma)
  → DO(conversationId).hydrate()
     → carga contexto relevante de Vectorize
     → llama Claude con prompt cacheado (1h)
     → si Claude invoca tool:
         Worker ejecuta → devuelve resultado a Claude
     → DO persiste nuevo estado + mensaje
  → respuesta outbound por WA API
  → Analytics Engine registra latencia, tokens, satisfacción
Trade-offs. DO por conversación da strong consistency y estado rico, pero over-engineered si hay menos de ~100 conversaciones concurrentes; en ese caso basta KV + Worker. El prompt caching es la diferencia entre $0.02 y $0.0003 por turno — impacta el modelo de negocio.
02

SaaS multi-tenant con código custom por cliente

Cada cliente (hotel, clínica) quiere lógica específica — descuentos, horarios, templates de mensajes, reglas de aprobación — pero mantener un fork por cliente no escala y obliga a un equipo de integración infinito.

Componentes
Dispatcher Worker Workers for Platforms · dispatch namespace DO Facets · SQLite por tenant Cloudflare for SaaS · custom hostname Access · policy por tenant Analytics Engine central
Flujo
Request → hostname tenant (acme.eigen.dev)
  → Dispatcher Worker resuelve tenantId por hostname
  → invoca Worker aislado en namespace del tenant
     → usa DO Facet con SQLite exclusiva del tenant
     → puede ejecutar código custom (subido por cliente o por Eigen)
  → respuesta al usuario
  → métricas homogéneas a Analytics Engine central
Trade-offs. Workers for Platforms es caro a baja escala y exige disciplina de runtime bounds. Si solo tienes 3-5 tenants, un solo Worker con tenantId como variable es más simple. El salto a esta arquitectura se justifica a partir de ~20 tenants con divergencia real.
03

Pipeline analítico edge-nativo

Quieres eventos capturados, datos históricos consultables, y dashboards — sin Kafka, sin Snowflake, sin Airflow, sin ingenieros de datos dedicados.

Componentes
Cloudflare Pipelines R2 Data Catalog · Iceberg Analytics Engine · near-realtime DuckDB / ClickHouse (ad-hoc) Workers · Chart.js dashboards Logpush · destinos externos
Flujo
Eventos HTTP (clicks, webhooks, conversions)
  → Pipeline ingesta (buffer + batch + transform)
     ├─→ R2 Data Catalog (parquet + Iceberg meta)
     └─→ Analytics Engine (métricas near-realtime)

Queries:
  - Analytics Engine (SQL) → Worker → dashboard HTML
  - R2 Data Catalog → DuckDB/ClickHouse → reportes ejecutivos
Trade-offs. Analytics Engine es perfecta para métricas dimensionales a alta cardinalidad (latencia por endpoint por tenant), pero no para joins complejos. R2 Data Catalog es excelente para analítica histórica pero agrega latencia de minutos. Combinarlas es la respuesta correcta.

App-per-usuario generada por IA (v0-style)

Usuarios describen en lenguaje natural una mini-app (dashboard, formulario, calculadora) y la quieren funcionando con persistencia propia, sin devs involucrados y sin compartir datos entre usuarios.

Componentes
Claude API · code generation Dynamic Workers DO Facets · SQLite por app Sandboxes · preview aislado R2 · assets estáticos Access · auth de usuario
Flujo
User describe app → Worker envía prompt a Claude
  → Claude genera código TS + schema SQLite
  → Worker deploya via Dynamic Workers (isolate aislado)
     → inicializa DO Facet con schema
  → preview en Sandbox (URL temporal)
  → al confirmar: publica a subdominio del user
  → Access garantiza que solo el owner entra
Trade-offs. Dynamic Workers + Facets (ambos beta) son la nueva primitiva ganadora. Antes habría requerido contenedores + DB per tenant + routing manual. La trampa: governance de código generado — necesitas Sandboxes y límites de CPU para evitar que un prompt hostil te haga bitcoin mining.
05

Backend híbrido Cloudflare + Postgres

Tu cliente ya tiene Postgres (Supabase, Neon, RDS) y no lo va a migrar. Tú quieres la velocidad edge de Workers sin pagar la latencia de conexión a Postgres en cada request.

Componentes
Worker API Hyperdrive · connection pool edge Supabase / Neon / RDS D1 · cache local Queues · writes lentos Smart Placement
Flujo
Request read:
  Worker → D1 cache (hit ratio ~70%)
     └─→ miss: Hyperdrive → Postgres → respuesta → cachea en D1

Request write:
  Worker valida → Queue (dedupe id)
     → Consumer Worker → Hyperdrive → Postgres
     → invalida D1 cache del recurso
     → notifica al usuario vía WS o email
Trade-offs. Hyperdrive elimina cold starts de pool pero no la latencia física al datacenter Postgres; Smart Placement ayuda colocando el Worker cerca. D1 como cache requiere disciplina de invalidación. Para tablas con high write frequency, el cache se vuelve ruido.

Intake inteligente de correo

Clientes reciben emails mezclados: citas, quejas, pagos, spam, facturación. Un humano clasifica y responde, o se pierden oportunidades por volumen.

Componentes

Flujo
Email entrante → MX de Cloudflare → Email Routing
  → forward a Email Worker
     → parse MIME (texto, attachments)
     → Claude clasifica: {agendar, cobrar, queja, spam, urgente}
     → Workflow ejecuta acción:
         - agendar: crea cita en Google Calendar, responde confirmación

Plataforma AI con MCP como superficie pública

Quieres que tus clientes (y los agentes que ellos usan — Claude Desktop, Cursor, Copilot) puedan acceder a los datos y acciones de Eigen desde sus propias herramientas, sin construir UI por cada integración.

Componentes
MCP Server Worker Workers OAuth Provider Access · OAuth gating API Shield · schema validation Rate limiting por tenant AI Gateway · observability
Flujo
Cliente abre Claude Desktop → agrega mcp.eigen.dev como tool
  → OAuth flow: MCP redirige a Access/OAuth Provider
  → cliente autoriza con credenciales de Eigen
  → token emitido con scopes del tenant
Claude invoca tool:
  → request a MCP Worker con Bearer token
  → API Shield valida schema

08

Observabilidad agent-native

Los agentes no fallan con stack traces: fallan dando una respuesta subóptima. La observabilidad tradicional (Sentry, logs) no captura "la respuesta fue correcta pero mala", y sin medir eso no hay mejora sistemática.

Componentes
Workers productivos Tail Worker Analytics Engine · métricas Logpush → R2 · audit trail AI Gateway · prompt logging Sentry · errores humanos Slack · alertas críticas
Flujo
Worker productivo emite eventos estructurados
  → Tail Worker recibe en vivo
     ├─→ métricas (latencia, tokens, tool-call success) → Analytics Engine
     ├─→ prompts/responses completos → AI Gateway + Logpush a R2
     ├─→ errores con stack → Sentry
     └─→ alertas (bad rating, tool fail crítico) → Slack

Eval loop offline:
  R2 → DuckDB → queries sobre historia → casos borderline → eval suite
Trade-offs. Logpush completo de prompts/responses es expensive y levanta flags de privacidad con datos médicos. Solución: Logpush con PII redaction + retención corta (7 días) + sampleo para eval dataset.
09

Sistema de pagos confiable con idempotencia

Los webhooks de Stripe llegan duplicados, fuera de orden, tarde. Procesarlos ingenuamente causa cobros dobles, emails repetidos, estados inconsistentes. Los incidentes de pago son los que menos toleran los clientes.

Componentes
Stripe webhook endpoint Worker verificador Queue · dedupe por event.id Consumer Worker DO por customerId · estado autoritativo Workflow · dunning Email Workers · notificaciones
Flujo
Stripe → POST webhook → Worker verifica firma HMAC
  → encola evento con message_id = stripe.event.id (dedupe nativo)
  → devuelve 200 inmediatamente (Stripe no reintenta)

Consumer Worker procesa con at-least-once:
  → hydrate DO(customerId)
  → DO aplica evento idempotentemente:
       if event.id in DO.processed_events: skip
       else: update state, push to processed_events
  → según tipo: trigger Workflow de dunning (pagos fallidos),
               o Email Worker con recibo (pagos exitosos)
  → escribe métrica a Analytics Engine
Trade-offs. DO es single-writer por customerId — correcto para consistency, pero cuidado con "hot customers" (uno que genera 1000 eventos/segundo). Para esos casos, particionar por event_type dentro del DO o usar un DO secundario.
10

Zero-trust internal tools

Dashboards de operación, admin panels, herramientas internas de QA — todo corre en subdominios que no deberían estar en el internet público, pero tampoco quieres montar VPN o un BastionHost.

Componentes
Cloudflare Access · SSO Workers (internal tools) Workers VPC + Mesh Hyperdrive · DB privada Gateway · filtro DNS del equipo Audit logs a R2
Flujo
Usuario → admin.eigen.dev
  → Access intercepta: ¿email autorizado? ¿MFA? ¿device posture?
     - sí → emite JWT de sesión (15 min TTL)
     - no → deniega con log en audit trail
  → request pasa al Worker con headers firmados (Cf-Access-Authenticated-User-Email)
  → Worker aplica autorización fina (role-based)
  → via Workers VPC + Mesh accede a DB privada (no público)
  → responde datos + escribe audit log a R2
Trade-offs. Access es notablemente más fácil que Auth0 para cuentas chicas, pero el vendor lock-in es real — migrar a otro IdP más adelante implica re-implementar. Para Eigen en etapa temprana es la elección correcta; revisar cuando haya >50 empleados.

Agente de voz en tiempo real

El paciente o huésped no quiere escribir — quiere hablar. Y el agente necesita responder en menos de 800 ms con la voz del dominio (dra. X, recepción del hotel) para no sentirse robótico.

Componentes
Workers Calls · WebRTC SFU Worker audio pipeline Whisper (en Containers) Claude · streaming SSE ElevenLabs TTS DO sesión de voz AI Gateway Guardrails
Flujo
User abre llamada → Workers Calls establece WebRTC
  → pcm chunks llegan al Worker cada 200 ms
     → VAD (voice activity detection) en edge
     → cuando detecta fin de frase:
         Container con Whisper transcribe → texto
         → Guardrails filtra PII/injection
         → Claude API con streaming (SSE) genera respuesta
         → tokens salen a ElevenLabs TTS streaming
         → audio chunks vuelven por Workers Calls al user
  → DO(sessionId) persiste turn history + resumen cada 10 turnos
Trade-offs. La latencia p95 es implacable: Whisper ~300 ms, Claude TTFB ~400 ms, TTS ~200 ms = ~900 ms antes del primer fonema. Paralelizar TTS con Claude streaming y precomputar saludos ayuda. Para <500 ms real-time, considerar modelos especializados tipo GPT-4o voice o Gemini 2.0 Live.

Streaming LLM con SSE / WebSockets

Una respuesta del LLM de 400 tokens tarda ~4 s. Hacer al usuario esperar 4 s con spinner mata la percepción de producto; mostrar los tokens a medida que llegan lo transforma.

Componentes
Worker endpoint SSE Claude API · stream: true ReadableStream + TransformStream ctx.waitUntil · logging async AI Gateway · cost tracking DO · si hay multi-cliente
Flujo
Client → GET /chat (Accept: text/event-stream)
  → Worker crea ReadableStream
  → inicia llamada Claude con stream: true
     → cada chunk de Claude → TransformStream
       → reformatea como SSE "data: {text}\n\n"
       → escribe al ReadableStream que va al cliente
  → al finalizar:
     ctx.waitUntil(persistirHistorial + logging a Analytics Engine)
  → cliente va pintando tokens en React con EventSource API
Trade-offs. SSE es simple y funciona tras proxies; WebSockets requieren más setup pero permiten bidireccional (tools callbacks). Para chat unidireccional (user pregunta, bot responde), SSE gana por simplicidad. El ctx.waitUntil es lo que hace esto correcto — persistir no debe bloquear la respuesta del último token.

Multi-region active-active con failover

Un cliente enterprise exige SLA 99.99% y tolerancia a caída completa de una región cloud. Workers está globalmente distribuido, pero tu DB no lo está.

Componentes
Workers (stateless, globales) Load Balancing · health checks D1 read replicas Postgres primario (us-east) + réplica (eu-west) Hyperdrive por región Queues · eventual sync Monitors + Slack alerts
Flujo
Request → Cloudflare Edge (anycast, PoP más cercano)
  → Worker decide read vs write:
     - read: intenta D1 replica local; miss → Hyperdrive a réplica Postgres local
     - write: siempre a primario vía Hyperdrive us-east
  → LB monitorea health de Postgres primario
     - si falla: promueve réplica + redirige Hyperdrive (manual o auto)
     - Queues acumula writes fallidos para replay al recovery

Fallback staged:
  nivel 1: servir desde D1 replicas (read-only degradado)
  nivel 2: servir página de mantenimiento (Cloudflare Pages)
Trade-offs. Active-active real requiere consistencia eventual aceptada por el producto. Para finanzas (pagos, balances) no aplica — mejor single-master con failover rápido. Para lectura-dominante (catálogos, búsquedas), active-active paga. La complejidad operacional se triplica; reservar para cuando el SLA lo justifique económicamente.

Colaboración en tiempo real con CRDTs sobre Durable Objects

Múltiples usuarios editan el mismo recurso (agenda del hotel, expediente clínico colaborativo, checklist de turno) y las ediciones deben mergearse sin conflictos ni overwrite ciego.

Componentes
DO por documento WebSockets + Hibernation Yjs / Automerge CRDT R2 · snapshots periódicos Access · autorización por documento
Flujo
User abre documento → WebSocket al DO(docId)
  → DO mantiene estado CRDT (Yjs doc) en memoria
  → cada edición local → update CRDT → broadcast a todos los WS del DO
  → DO aplica update a su copia (conmutativo, siempre converge)
  → periódicamente (cada 30s o 100 ops):
     DO snapshot → R2 (persistencia durable)
  → al hibernar: estado se restaura del último snapshot + eventos R2

Concurrencia: Yjs maneja merge automático, no hay conflictos ni OT manual
Trade-offs. DO single-threaded es el encaje perfecto para CRDT colaborativo — sin locks, sin conflictos. El límite: un DO por documento escala bien hasta ~200 colaboradores simultáneos; más requiere particionar por secciones. Para Eigen, vale si algún cliente (hotel con múltiples recepcionistas) realmente necesita colaboración en vivo; probablemente over-engineering para la mayoría.

Feature flags + experimentación A/B

Quieres activar features por cliente, por región, por porcentaje de tráfico, y medir el impacto — sin redeploys y sin tocar código de producto.

Componentes
KV · flags globales Worker · evaluator DO · sticky bucketing por user Analytics Engine · exposiciones + conversiones Admin UI con Access Versioned Deploys (opcional)
Flujo
Request llega con userId
  → Worker carga flags de KV (cached local ~1 ms)
  → para cada flag:
     if targeting "por_cliente": check tenant
     if targeting "por_porcentaje":
        bucket = hash(userId + flagKey) % 100
        si bucket < rollout → ON
     DO(userId).recordExposure(flagKey, variant)
  → Worker ejecuta con flags resueltos
  → eventos de conversión llegan → correlación con exposures
  → dashboard: lift por variant + significance testing
Trade-offs. Sticky bucketing con hash(userId + flagKey) es determinístico y no requiere DO — solo el recording sí. Alternativa simple si no necesitas per-user targeting: Versioned Deploys con Gradual Rollout hace A/B a nivel deploy. Para experimentos de producto (UI), los flags son correctos; para cambios infraestructurales, usar rollouts.
16

Webhook fan-out / event bus interno

Un evento (pago confirmado, cita agendada, lead nuevo) debe disparar N acciones: email, actualización de CRM, notificación interna, actualización de analytics, trigger de Zapier. Hacerlas sincrónicamente es frágil.

Componentes
Worker publisher Queue por tipo de evento N Consumer Workers (uno por handler) Service Bindings internos Dead-letter Queue Logpush · audit trail
Flujo
Evento origen (pago, webhook, acción manual)
  → Worker publisher valida + normaliza schema
  → escribe a Queue("payment.confirmed") con metadata común
  → Cloudflare entrega a N consumers configurados:
     - consumer_email → envía recibo al cliente
     - consumer_crm → actualiza HubSpot via API
     - consumer_analytics → escribe a Analytics Engine
     - consumer_internal → notifica a Slack
  → cada consumer es independiente:
     falla uno → dead-letter queue, reintento, no afecta otros
  → Logpush guarda audit completo en R2 para replay
Trade-offs. Queues en CF son at-least-once con dedupe opcional — los consumers deben ser idempotentes. Fan-out vía Service Bindings (invocación directa) es más rápido pero acopla deploys; vía Queues desacopla y resiliente, pero agrega ~50-200 ms de latencia. Para eventos user-facing no críticos, Queues gana; para flujos user-blocking, Service Bindings.

API aggregator con rate limiting por cliente

Expones varias APIs internas bajo una surface unificada (para partners, integradores, otros clientes de Eigen), con planes distintos (free, pro, enterprise) y cada uno con cuotas distintas por endpoint.

Componentes
Worker gateway API Shield · schema validation Rate Limiting por API key Workers OAuth Provider · issuing DO por api_key · cuotas + usage Service Bindings a APIs internas Logpush + Analytics Engine
Flujo
Request de partner → gateway con Bearer token
  → Worker valida token (OAuth Provider)
  → API Shield valida body contra schema OpenAPI
  → DO(apiKey).consume(endpoint, cost):
     - revisa plan + cuota mensual
     - decrementa balance
     - si excede → 429 con Retry-After
  → Service Binding a API interna (Billing, Calendar, Patients)
  → respuesta con headers X-RateLimit-Remaining
  → Logpush escribe cada call con costo → Analytics Engine
Trade-offs. Rate Limiting nativo de CF es más simple y barato que un DO por key, pero el DO permite cuotas mensuales acumulativas y facturación por uso — imposible con el rate limiter built-in. Usar ambos: rate limiter para burst protection (por segundo), DO para cuota (por mes).
18

Pipeline de content moderation

Usuarios suben fotos (perfil de paciente, tickets de reservación, evidencia de queja) o texto abierto (reviews, chat público). Cualquiera puede ser PII, abuso, contenido ilegal o daño reputacional si se almacena y se reindexa.

Componentes
Worker ingress Cloudflare Images · intake Workers AI · Llama Guard · CLIP Claude · second-pass review R2 · quarantine + published buckets Queue · async pipeline Workflow · human review
Flujo
Upload llega al Worker → tipo-detect
  Texto:
    → Workers AI Llama Guard clasifica toxicidad/PII
    → si borderline → Claude segunda pasada con contexto
    → score > umbral → quarantine R2; score limpio → R2 público
  Imagen:
    → Cloudflare Images recibe (thumbnail auto)
    → Workers AI CLIP clasifica (NSFW, violencia, PII visible)
    → si detecta cara: opcional face-blur automático
    → quarantine vs publish igual que texto
  Queue → Workflow de revisión humana si borderline
    → reviewer humano aprueba/rechaza → decisión persistida
  Todo el pipeline genera audit log inmutable en R2
Trade-offs. Llama Guard es barato y suficiente para el 90% del volumen; Claude como second-pass agrega costo pero salva el 10% borderline. Face-blur automático para fotos de pacientes puede ser buena decisión default por cumplimiento LFPDPPP — mejor pedir perdón que permiso legal. Archivos rechazados van a bucket encriptado con retención de 30 días por posible disputa.
