§00

Por qué Cloudflare ahora

Hace cinco años era infraestructura de fondo. Hoy es la plataforma donde se construyen productos.

Cloudflare empezó como red de distribución de contenido — velocidad de carga, protección DDoS, DNS. Durante una década fue eso: una capa que aceleraba lo que ya existía. En 2018 lanzó Workers, y la plataforma empezó a cambiar. En 2024, con Workers AI y Vectorize, cambió de forma. En 2026, con Durable Object Facets, Agents SDK, Browser Run y Agent Lee, cambió de naturaleza.

Hoy Cloudflare no es infraestructura en el sentido clásico. Es una plataforma de aplicación en la que los servicios primitivos — compute, estado, red, modelos de IA — se componen con una coherencia que AWS, GCP y Azure nunca tuvieron porque crecieron por adquisición, no por diseño. Un Worker llama a un Durable Object llama a Workers AI llama a Vectorize, todo en la misma red, con el mismo lenguaje de bindings, con latencia intra-PoP medida en microsegundos.

Para nosotros — constructores de productos B2B en 2026 — esto importa por tres razones:

  • Velocidad de ejecución. Lo que hace cinco años requería EC2 + RDS + ElastiCache + Lambda + SQS + SNS + IAM + CloudWatch, hoy es Workers + D1 + KV + Queues + Analytics Engine — con una sola configuración, una sola facturación, un solo mental model.
  • Costos predecibles. No hay sorpresas de egress. No hay sorpresas de NAT gateway. Los pricing tables son legibles por humanos. En Eigen, donde vendemos margen sobre costo, esto es oro.
  • Diseño AI-nativo. Cloudflare es la primera plataforma cuya superficie está diseñada para que agentes de IA operen sobre ella. El MCP oficial de Cloudflare da a Claude Code y a Cursor acceso inmediato a todo el API. Agent Lee es literalmente un agente de IA que vive en el dashboard. Esto no existe todavía en AWS.
Bajo el lente de IA

La consecuencia práctica es que tú no aprendes Cloudflare solo. Lo aprendes con Claude Code. Cada vez que estás atorado, le preguntas. Cada vez que quieres probar algo, lo deployas con cf deploy desde una conversación. El manual de la plataforma está indexado en el MCP — Claude lo consulta en tiempo real. Si sabes articular tu intención con precisión, construyes a velocidad de pensamiento.

Pero eso no reemplaza entender los fundamentos. El objetivo de esta clase no es que puedas pedirle a Claude "hazme un Worker" — es que puedas leer el código que Claude produjo, detectar el silent failure que introdujo, y articular por qué la arquitectura propuesta no va a escalar. Entender sin necesidad de IA es lo que te permite usar la IA con criterio.

§01

La filosofía: edge, Anycast, isolates

Para construir bien sobre Cloudflare, primero entiende en qué se diferencia de todo lo demás.

Cloudflare opera una red de ~330 data centers (llamados PoPs — points of presence) en ~120 países. Cuando un usuario hace una solicitud HTTP a un dominio servido por Cloudflare, la solicitud llega al PoP más cercano físicamente al usuario — no al servidor de origen. Esto no es "cloud computing" en el sentido de AWS us-east-1. Es edge computing: el cómputo vive en el perímetro de la red, cerca del usuario.

Anycast

La dirección IP de tu Worker en Cloudflare no es una dirección específica de un servidor. Es una dirección anycast: muchos servidores en muchos países anuncian la misma IP al internet, y el protocolo BGP enruta cada request al servidor más cercano al emisor. Es por esto que cloudflare.com responde en 15ms desde Ciudad de México y también en 15ms desde Berlín — son dos PoPs distintos respondiendo al mismo dominio.

V8 isolates: el corazón de Workers

Aquí está la idea que hace que Workers exista: los contenedores son demasiado pesados para edge computing. Un contenedor Docker tarda segundos en arrancar, consume megabytes de memoria en overhead, y requiere un kernel Linux completo. Si cada request abriera un contenedor nuevo, el edge sería más lento que el origen.

Cloudflare toma prestado de los browsers una primitiva distinta: el V8 isolate. V8 es el motor de JavaScript que corre en Chrome. Un isolate es una instancia aislada del runtime dentro del mismo proceso — no es un contenedor, no es una VM, es una región de memoria con su propio heap y sus propios globales. Arranca en menos de 5 milisegundos, consume cientos de kilobytes, y miles de isolates coexisten en el mismo proceso con aislamiento fuerte entre ellos.

La consecuencia crítica Cold starts no existen en Workers. Tu código se compila la primera vez que se despliega, y a partir de ahí, cada request a cualquier PoP puede ejecutarlo sin arrancar nada. Esto cambia cómo diseñas: puedes tener Workers "caros" de ejecutar sin temor a penalizar al primer usuario.

La consecuencia de diseño: sin procesos

Como Workers corre en isolates y no en procesos, no puedes tener un servidor HTTP escuchando entre requests. No hay "background jobs" en el Worker mismo. No puedes escribir a disco porque no hay disco. Esto suena como limitación pero es liberador: te empuja a usar las primitivas correctas — Durable Objects para estado persistente por entidad, Queues para trabajo async, Workflows para pipelines de larga duración, Cron Triggers para tareas programadas.

Bajo el lente de IA

Cuando Claude Code genera un Worker, a veces importa paquetes de Node que asumen un runtime completo — fs, child_process, http.createServer. Si no entiendes esta diferencia, aceptas el código y fallas en deploy. Si la entiendes, identificas el error conceptual y corriges el rumbo: "Claude, reescribe esto usando fetch y R2, no Node primitives."

§02

Workers: el compute primitivo

La unidad fundamental sobre la que todo lo demás se compone.

Un Worker es una función TypeScript (o JavaScript, WebAssembly, Python vía Pyodide) con un handler principal — típicamente fetch — que Cloudflare ejecuta en respuesta a un request HTTP. La forma más simple posible:

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext): Promise<Response> {
    return new Response("Hola desde el edge.");
  }
};

Los tres parámetros

El handler recibe tres argumentos. Entender cada uno es entender Workers:

  • request — la solicitud HTTP entrante, idéntica al API Request del estándar web. Contiene request.cf con metadata de Cloudflare (colo, país, bot score, TLS version, etc.).
  • env — el environment. Contiene tus bindings declarados en wrangler.toml: acceso a KV, D1, R2, DO, Queues, secrets. Esto es cómo los Workers hablan con el resto de Cloudflare. Un binding no es una URL, es una referencia tipada que Cloudflare inyecta en runtime.
  • ctx — el execution context. Ofrece dos primitivas críticas: ctx.waitUntil(promise) para trabajo async que debe terminar después de responder (logging, analytics, cache warming); y ctx.passThroughOnException() para que si el Worker falla, el request pase directo al origen como fallback.

Handlers más allá de fetch

Workers soporta varios tipos de handler, cada uno con su propio contexto:

  • fetch — request HTTP (el común).
  • scheduled — Cron Trigger. Se ejecuta en horario configurado.
  • queue — consumer de una Queue. Recibe batch de mensajes.
  • email — Email Worker. Recibe correo entrante como MIME parseado.
  • tail — Tail Worker. Recibe eventos de ejecución de otros Workers (logs, traces, errores).
  • alarm — dentro de un Durable Object, despierta cuando la alarma programada dispara.

Bindings: el modelo de composición

En lugar de hacer fetch("https://kv.example.com/...") con credenciales, el Worker hace env.MY_KV.get("key"). El binding es una referencia que Cloudflare resuelve en runtime, y está tipada — TypeScript sabe que env.MY_KV es un KVNamespace. Esto tiene dos consecuencias:

  1. Zero network overhead. Entre Workers del mismo account y recursos bindeados, no hay HTTP real; es una llamada en memoria al binding layer.
  2. Tipado de punta a punta. Si cambias el schema de un D1, tu env.MY_DB refleja el cambio después de regenerar tipos. Catches de error en compile time, no en runtime.
Bajo el lente de IA

Cuando generas un Worker con Claude Code, una de las primeras cosas que hace es preguntar (o asumir) qué bindings necesita. Un prompt bien engineerado menciona explícitamente: "este Worker necesita binding a D1 llamado DB, KV llamado CACHE, y Queue llamada JOBS". Claude genera el wrangler.toml correcto y los tipos consistentes. Si no eres explícito, Claude infiere, y ahí aparecen inconsistencias sutiles entre el código y la configuración.

§03

Estado: KV, D1, R2, Durable Objects

Cloudflare tiene cinco maneras distintas de guardar datos. Elegir la correcta es oficio.

Un Worker es stateless — cada request es una invocación independiente. Para guardar estado hay cinco primitivas, cada una con semántica específica. La habilidad distintiva de un Solutions Architect no es conocerlas — es saber cuál elegir para cada caso, con números en mano.

Workers KV · eventual consistency global

KV es un diccionario clave-valor globalmente distribuido. Escribes una clave en cualquier PoP, y la escritura se propaga a todos los PoPs del mundo en 60 segundos o menos. Lecturas son locales — leen del PoP más cercano al usuario, con latencia típica <10ms.

Su ventaja: lecturas rápidas globales. Su costo: writes son eventualmente consistentes. Si escribes KV.put("balance:u123", "100") y un microsegundo después lees KV.get("balance:u123") desde otro PoP, puede que todavía veas el valor viejo.

Cuándo usar: configuración, feature flags, sesiones (con TTL corto), cache de respuestas HTTP pesadas. Cuándo no usar: contadores, balances, cualquier cosa donde la consistencia importa.

D1 · SQLite en el edge

D1 es una base de datos relacional SQLite que Cloudflare replica en el edge. Soporta SQL completo — joins, window functions, CTEs, transacciones. Desde 2026 tiene read replicas en múltiples regiones y una Sessions API para lecturas consistentes dentro de una transacción lógica.

D1 es para datos estructurados que caben en una base relacional: tablas de usuarios, reservas, catálogos de producto. Su límite: no es Postgres. Si tu app necesita volumen transaccional serio, joins complejos a escala, o extensiones tipo PostGIS, usas Postgres externo vía Hyperdrive (que veremos).

R2 · blob storage sin egress

R2 es el S3 de Cloudflare. Compatible con la API S3, sin fees de egress (descargar datos no cuesta). Soporta URLs firmadas, event notifications (dispara Workers cuando un objeto cambia), y desde 2026 tiene R2 Data Catalog — una capa de Apache Iceberg para queries analíticas directamente sobre R2 con DuckDB, ClickHouse o Spark.

Úsalo para archivos — imágenes, videos, PDFs, checkpoints de modelos, logs archivados, snapshots de backup. No lo uses para datos estructurados pequeños que cambien frecuentemente (eso es D1 o KV).

Durable Objects · estado consistente por entidad

DO es la primitiva más distintiva de Cloudflare y también la más difícil de entender al principio. Un DO es un actor: un objeto identificado por un ID (típicamente un userId, conversationId, hotelId) que tiene estado en memoria + estado persistente, y que garantiza que solo una instancia del DO existe globalmente para un ID dado. Cloudflare migra la instancia al PoP donde más se usa.

Esto es mágico para casos donde necesitas strong consistency por entidad: contadores por usuario, sesión de chat por conversación, estado de reserva por huésped. No hay race conditions entre requests al mismo DO — son serializados automáticamente. No necesitas locks, no necesitas optimistic concurrency, no necesitas nada. El actor lo resuelve.

Además, DO tiene hibernation: si no hay actividad por un tiempo, Cloudflare lo hiberna. Cuando llega un request, se despierta con su estado intacto. No pagas por DOs idle.

Desde abril 2026, DO tiene Facets: cada instancia puede tener su propia base SQLite aislada. Esto habilita multi-tenant por defecto — un DO por cliente, cada uno con sus propios datos, sin pre-provisionamiento.

Cache API · HTTP cache programable

La Cache API (caches.default) es el cache HTTP por PoP de Cloudflare, accesible desde Workers. Es cache efímero, regional (no global como KV), y barato. Úsalo para responses HTTP caras que cambian poco y que pueden regenerarse sin problema.

Claude Code tiende a elegir DO por defecto porque "DO es flexible". Es sobrediagnóstico: si solo necesitas guardar config, un DO es overkill — KV resuelve con menos complejidad. §04

Orquestación: Queues, Workflows, Service Bindings

Cómo se coordinan las piezas cuando el trabajo no cabe en un solo request.

Un request HTTP en Workers tiene un budget limitado — típicamente hasta 30 segundos de CPU. Para trabajo que excede ese presupuesto, o que debe sobrevivir reinicios, Cloudflare ofrece tres primitivas de orquestación cuya combinación resuelve la mayoría de sistemas productivos.

Queues · async confiable

Queues es la primitiva de mensajería. Un productor escribe mensajes, un consumer los procesa. Garantías: at-least-once delivery (un mensaje puede entregarse más de una vez — tu consumer debe ser idempotente), batching configurable (el consumer recibe hasta N mensajes por invocación), retries con backoff automático, dead-letter queue para mensajes que fallan repetidamente.

Modelo mental correcto: Queues es el reemplazo de Kafka/SQS para la mayoría de casos. No tiene ordering global (los mensajes pueden llegar fuera de orden entre batches), pero tiene ordering por consumer batch y dedupe opcional.

// Productor: en un Worker fetch handler
await env.JOBS.send({ type: "send_email", userId: "u123" });

// Consumer: en un Worker queue handler
export default {
  async queue(batch: MessageBatch, env: Env) {
    for (const msg of batch.messages) {
      try {
        await processJob(msg.body);
        msg.ack();
      } catch (err) {
        msg.retry({ delaySeconds: 60 });
      }
    }
  }
};

Workflows · ejecución durable

Workflows es la evolución de "cron + scripts". Defines una serie de steps; Cloudflare los ejecuta con persistencia automática del estado entre ellos. Si un step falla, reintenta con backoff. Si el Worker muere, Cloudflare retoma el workflow donde quedó. Cada step tiene su propio timeout. Puedes programar sleeps de días, semanas, meses.

Desde 2026 tiene features clave: signals externos (un workflow puede esperar a que otro sistema lo despierte — perfecto para human-in-the-loop), y compensación estilo saga (si el paso 5 falla, ejecuta la compensación de los pasos 1-4 para deshacer).

El patrón que lo transforma todo El flujo clásico de "cron job en EC2 que corre cada hora y procesa una cola" se vuelve: Workflow iniciado una vez, con step "fetch trabajo", step "procesar", step "notificar", step "sleep 1 hora", loop. No hay infraestructura que monitorear. No hay "¿por qué el cron no corrió?". Si Cloudflare está arriba, tu workflow sigue ejecutándose.

Service Bindings · invocación sin red

Un Worker puede llamar a otro Worker sin ir por HTTP, usando un binding directo. La llamada no cruza la red — es una invocación en memoria entre isolates, con latencia de microsegundos, tipada, sin costo adicional.

Esto habilita un patrón arquitectónico poderoso: microservicios lógicos (código modular, responsabilidades separadas, repos independientes) sin el costo de microservicios físicos (HTTP interno, service mesh, latencia). Tu Worker "billing" es un módulo; tu Worker "agenda" es otro módulo; ambos se componen en un tercero via bindings.

Cron Triggers y DO Alarms

Cron Triggers ejecutan un Worker en horarios configurados (sintaxis cron estándar). Hasta 1,000 crons por account, sin infraestructura. Para scheduling por-entidad (un timer por usuario, por cita, por pedido), usas DO Alarms — un DO programa una alarma, Cloudflare dispara el handler cuando es tiempo.

Bajo el lente de IA

Cuando pides a Claude un pipeline de procesamiento, su primer instinto es escribir una función síncrona gigante. Detenlo: "Usa Workflows con steps atómicos, y marca los pasos críticos como idempotentes." El código resultante es infinitamente más robusto — pero solo si tú sabes pedirlo. Este es el patrón recurrente de trabajar con Claude bien: la IA escribe rápido, pero tú le das la estructura correcta.

§05

La capa de IA: modelos, Vectorize, AI Gateway

Cloudflare no entrena modelos. Los sirve, los memoriza, los enruta y los factura con inteligencia.

Cloudflare tiene tres servicios principales para construir productos de IA: Workers AI (inferencia hosted de modelos open-source), Vectorize (base de datos vectorial para memoria semántica), y AI Gateway (plano de observabilidad y control para llamadas a cualquier modelo). Estos tres componen la capa AI-nativa junto con Claude API (externa) y el Agents SDK (la veremos en §06).

Workers AI · inferencia en el edge

Cloudflare hospeda un catálogo creciente de modelos open-source: Llama, Mistral, Gemma, CLIP, Whisper, Stable Diffusion, embeddings. Los invocas desde un Worker con un binding:

const response = await env.AI.run(
  "@cf/meta/llama-3.1-8b-instruct",
  {
    messages: [{ role: "user", content: "Resume este texto: ..." }]
  }
);

La inferencia corre en GPUs distribuidas en los PoPs. Latencia típica: 200–800 ms para modelos pequeños. Desde 2026 soporta structured outputs (JSON schema), tool calling nativo, y contextos más largos. GPT-5.4 fue agregado al catálogo en abril.

Workers AI es perfecto para: clasificación, moderación, embeddings, transcripción, generación de imágenes simples. No es perfecto para: razonamiento pesado de frontera (ahí gana Claude), fine-tuning custom (ahí va a Modal), latencia sub-100ms (ahí ganan modelos edge específicos).

Vectorize · memoria semántica

Vectorize es una base de datos vectorial. Guardas embeddings (vectores numéricos que representan semánticamente un documento o frame de imagen) junto con metadata, y consultas por similitud. Soporta filtros por metadata, namespaces para multi-tenancy, y escala a millones de vectores sin operación propia.

// Insertar embedding
await env.INDEX.insert([{
  id: "doc-42",
  values: embedding,       // array de 1536 floats
  metadata: { tenantId, category: "menu" }
}]);

// Consultar por similitud
const results = await env.INDEX.query(queryEmbedding, {
  topK: 10,
  filter: { tenantId }
});

Caso típico en Eigen: memoria de un agente conversacional — cada mensaje relevante se embebe y se guarda; cuando el paciente regresa, el agente recupera los 5 más relevantes al tema actual y los inyecta en el prompt.

AI Gateway · observabilidad y control de LLMs

AI Gateway es un proxy configurable delante de cualquier API de LLM: Anthropic, OpenAI, Google, Workers AI. Haces tus llamadas vía el gateway en lugar de directamente al proveedor. A cambio obtienes:

  • Caching automático de responses idénticas o similares (baja costo 30–70%).
  • Rate limiting y circuit breakers por modelo o por cliente.
  • Logging estructurado de prompts y respuestas (con redacción de PII opcional).
  • Costo atribuible por feature, usuario, o tenant.
  • Guardrails inline: detección de PII, prompt injection, jailbreak, toxicidad.
  • Fallback automático: si Anthropic responde 503, retry contra OpenAI.
Bajo el lente de IA

Sin AI Gateway, tu costo de producción se escapa y tu observabilidad es ciega. Con AI Gateway, el dashboard te dice: "este feature cuesta $0.02 por llamada, el 70% es prompt cache hit, el guardrail bloqueó 3 intentos de injection la semana pasada." Esto transforma la decisión de "¿cómo está funcionando mi agente?" de una pregunta filosófica a una pregunta con datos.

Patrón recomendado en Eigen: todas las llamadas a Claude o Workers AI pasan por AI Gateway, con logging opt-in para evals offline. El eval suite lee del log, no hace llamadas nuevas. Eso baja el costo de iterar evals 10× o más.

§06

Agentes: Agents SDK, Think, MCP

La diferencia entre un "bot con tool use" y un agente productivo vive en la arquitectura. Aquí está.

Un agente no es un LLM con un bucle de tool use. Es un sistema que persiste estado, ejecuta pasos, maneja fallos, aprueba acciones sensibles con humanos, y se observa a sí mismo. Cloudflare ofrece la plataforma más completa para construir esto en 2026.

Cloudflare Agents SDK · next edition (Think)

El SDK de Agents de Cloudflare, en su edición renovada lanzada el 15 de abril de 2026, redefine qué es un agente: cada agente es una clase TypeScript que corre sobre un Durable Object. El DO provee SQL propia, WebSockets, scheduler interno, memoria persistente. El framework Think se encarga de la persistencia — si el Worker muere, el agente retoma donde quedó.

export class MedicalIntakeAgent extends Agent<Env, State> {
  async onMessage(msg: string) {
    const context = await this.memory.retrieveRelevant(msg);
    const response = await this.llm.chat({
      system: this.systemPrompt,
      messages: [...context, { role: "user", content: msg }],
      tools: [this.scheduleTool, this.billingTool]
    });
    await this.memory.store(msg, response);
    return response;
  }
}

Features incluidos out-of-the-box: streaming chat, tools server-side y client-side, human-in-the-loop approval, task scheduling, memoria persistente. No reinventas nada de esto — lo compones.

Claude Agent SDK · la otra pieza

Del lado de Anthropic, el Claude Agent SDK define primitivas adicionales que combinas con el de Cloudflare: skills (unidades modulares con triggering por patrones y progressive disclosure), subagents (delegación a agentes especializados tipo coordinator/worker), hooks (interceptores pre/post tool use, Stop, SessionStart), y un modelo de memoria estructurada.

En la práctica, un agente Eigen combina ambos: la infraestructura (runtime, persistencia, scheduling) del Cloudflare SDK, y la disciplina (skills, subagents, hooks, memory) del Claude SDK.

MCP · Model Context Protocol

MCP es un estándar (iniciado por Anthropic, adoptado por Cloudflare, OpenAI, Microsoft) que define cómo un agente descubre y llama herramientas externas. Un servidor MCP expone tools; cualquier cliente MCP (Claude Desktop, Cursor, Copilot, Claude Code) puede conectarse y usarlas.

Cloudflare ofrece dos superficies:

  • Cloudflare MCP — servidor oficial que expone el API completo de Cloudflare (~3,000 operaciones) a cualquier cliente MCP. Tu Claude Code literalmente puede crear Workers, configurar DNS, provisionar D1 databases, desde un chat.
  • MCP servers hosteados sobre Workers — construyes tu propio MCP server para que tus tools de negocio sean accesibles a agentes externos. Con OAuth + rate limiting + schema validation. Tus clientes usan sus propios Claude Desktop o Cursor para interactuar con los servicios de Eigen.

Patrones multi-agente

Un agente monolítico con 20 tools es un anti-pattern. Los patrones correctos en 2026:

  • Coordinator / worker — un agente orquestador rutea requests a subagentes especializados (Scheduler, Billing, Medical Intake). Cada subagente tiene solo los tools que necesita.
  • Debate — dos agentes con system prompts opuestos discuten un caso; un tercero decide. Útil para decisiones reversibles donde el costo de error es alto.
  • Ensemble — misma pregunta a tres modelos distintos, respuestas agregadas por votación o síntesis. Mejora accuracy al costo de latencia y dinero.
  • Critic — después de que el agente responde, un crítico lo revisa antes de que salga al usuario. Detecta fallas obvias (alucinaciones, tono inapropiado, PII expuesta) con latencia adicional pequeña.
Bajo el lente de IA

Cuando Claude Code construye un agente por ti, lo que sugiere por default suele ser monolítico. Detente y pide: "Separa esto en subagents: uno para agenda, uno para pagos, uno para triage clínico. Cada uno con su propio system prompt corto. El coordinator los despacha según intent." El código resultante es significativamente más mantenible y testeable — y la accuracy sube porque cada subagent tiene contexto más enfocado.

§07

Red, delivery y media

El linaje original de Cloudflare. Sigue siendo la mitad del valor.

La historia de Cloudflare empieza con CDN. Después de 15 años, esa capa es la más madura, la más barata, y la más olvidada — hasta que la necesitas. Entender caching, delivery y media optimization te ahorra costos enormes y construye productos que se sienten rápidos.

CDN y Cache API

Cuando un request llega a un PoP, Cloudflare verifica si tiene el response en cache. Si sí, lo devuelve en milisegundos sin ir al origen. La cache key por default incluye URL + método + ciertos headers; puedes customizarla. TTL configurable, purge manual o por tag, Vary por headers para contenido personalizado.

La Cache API te da control programático desde Workers: puedes leer del cache, escribir al cache, evaluar condiciones, aplicar transformaciones. Esto habilita patrones tipo "stale-while-revalidate": devuelves contenido viejo rápido mientras refrescas en background con ctx.waitUntil.

Cache Reserve y Tiered Cache

Para contenido estático pesado (videos grandes, assets de gigabyte), Cache Reserve guarda el cache en R2 de forma permanente — aumenta hit ratio al 99%+ y reduce carga al origen a prácticamente cero. Tiered Cache crea jerarquía: PoP local → regional → origin. El regional ayuda cuando muchos PoPs cercanos pedirían lo mismo — consolida al regional antes de pegar el origen.

Cloudflare Images · optimización de imágenes

Images es storage + transformación on-the-fly. Subes una imagen original, pides variantes por URL (/image/abc/w=800,format=auto). Cloudflare devuelve WebP o AVIF o JPEG según el browser del usuario, con tamaño exacto pedido. URL signing para imágenes privadas.

En Eigen, cada landing de cliente y cada producto de hotel usa Images por default — no montamos Cloudinary, no corremos imgproxy, no pagamos por CDN separado.

Cloudflare Stream · video

Stream hace lo mismo para video: subes original, Cloudflare transcodifica automáticamente a HLS y DASH con adaptive bitrate, player embebido con DRM opcional, live streaming con latencia sub-segundo, recording. Analytics por viewer y segmento.

Workers Calls · WebRTC SFU

Workers Calls provee un Selective Forwarding Unit (SFU) para WebRTC a escala global — la primitiva para construir video/voz en tiempo real. Sin montar LiveKit, sin Jitsi propio, sin Twilio Video.

Browser Run (ex Browser Rendering)

Browser Run, renombrado y expandido el 15 de abril de 2026, da a tus Workers acceso a Chrome headless en el edge. Pero ya no es solo "headless Chrome como librería": tiene Live View (ver al agente operar en vivo), Human-in-the-Loop (cuando el agente topa con login o captcha, pasa control a humano que resuelve y devuelve), Session Recordings con rrweb-player, y acceso directo al Chrome DevTools Protocol.

Esto cambia qué es posible en automatización web. Tu agente puede navegar un portal bancario, ceder al humano cuando pide 2FA, y retomar. Sin infraestructura propia.

Bajo el lente de IA

Browser Run + HITL es el patrón que hace viable a agentes ejecutar acciones reales en portales que no tienen APIs. Antes requerías Puppeteer en un EC2, un sistema de handover manual, y rezar. Ahora es configuración declarativa. El cambio es significativo y la mayoría aún no lo ha internalizado — es una ventaja competitiva para quien sepa usarlo.

§08

Seguridad y Zero Trust

Cloudflare es, entre otras cosas, la empresa de seguridad más grande del mundo. No ignores esta capa.

El stack de seguridad de Cloudflare cubre desde protección DDoS a nivel de paquete hasta control de acceso aplicativo. Para Eigen, las capas más relevantes son WAF, Bot Management, Access, Tunnel, y las nuevas primitivas de agent security (Mesh, Workers VPC, Outbound Workers).

WAF · Web Application Firewall

WAF inspecciona cada request contra reglas managed (OWASP Top 10 cubierto por default) y reglas custom escritas en Wirefilter (un lenguaje tipo BPF para expresar condiciones sobre requests). Modos: log, simulate, block. Esto captura SQL injection, XSS, path traversal y ataques similares antes de que lleguen a tu Worker.

Bot Management

Cloudflare analiza cada request con ML entrenado sobre el tráfico agregado de toda su red. Cada request recibe un bot score (1–99). Tú decides qué hacer: bloquear bots hostiles, dejar pasar bots legítimos (Google, GPTBot), retar a los ambiguos con JavaScript challenges.

Cloudflare Access · SSO y zero-trust

Access es la alternativa de Cloudflare a Auth0/Okta para proteger aplicaciones internas. Configuras: "admin.eigen.dev solo acepta emails @eigen.dev con MFA obligatoria." Cloudflare intercepta cada request, valida el JWT de sesión, y solo deja pasar los autorizados. Auditoría completa, políticas por path, sesión con TTL configurable.

Para Eigen con pocos empleados, Access es notablemente más simple que montar Auth0 — setup de 20 minutos, $0 en el free tier, escalable.

Cloudflare Tunnel · expone servicios on-prem

Tunnel (el binario cloudflared) establece una conexión saliente encriptada desde tu infraestructura on-prem o en otra cloud, hacia la red de Cloudflare. Tus servicios internos quedan accesibles a través del edge sin abrir puertos inbound. Complementado con Access, obtienes zero-trust a cualquier servicio TCP/UDP sin VPN.

Workers VPC + Mesh (abril 2026)

Para agentes que necesitan acceso privado a bases de datos o APIs internas, Workers VPC + Mesh provee la superficie zero-trust correcta. Sin tunneles manuales, sin VPNs frágiles. Cloudflare maneja routing, authentication, y audit.

Outbound Workers para Sandboxes (abril 2026)

Cuando ejecutas código generado por IA en un Sandbox, no quieres darle tus credenciales. Outbound Workers es un egress proxy que corre en el runtime Workers — fuera del sandbox. El sandbox hace una request plana; el Outbound Worker inyecta credenciales transparentemente antes de reenviar upstream. El sandbox nunca ve los tokens.

Además: TLS interception con CA efímera por instancia (la clave privada nunca sale del sidecar), allow/deny lists con glob patterns, políticas dinámicas actualizables sin restart.

Bajo el lente de IA

Los Outbound Workers resuelven un problema real de la era de agentes: tu agente ejecuta código que tú no escribiste directamente (lo generó Claude, o lo escribió un subagent autónomo). Ese código no debería tener acceso a tus API keys de Stripe. Antes, esto requería sandboxing complejo con cargo cognitivo alto. Ahora es un outbound handler de 20 líneas. Construir agentes seguros ya no es privilegio de empresas con equipos de infosec — es una decisión configuracional.

§09

Datos y analítica edge-nativa

El stack analítico que reemplaza Kafka + Snowflake + Airflow + Datadog con primitivas más simples.

La historia del data stack de los últimos 10 años es acumulación: Kafka para streaming, Snowflake o BigQuery para warehouse, Airflow para orquestación, Datadog o Grafana para métricas. Cloudflare propone uno más simple — no necesariamente más potente, pero suficiente para el 90% de casos B2B en 2026.

Cloudflare Pipelines · ingesta

Pipelines (lanzado en 2026) ingesta eventos vía HTTP, los bufferea, los transforma, y los persiste a R2 o a R2 Data Catalog. Reemplaza Kafka + Firehose con una superficie declarativa. Sin ZooKeeper, sin cluster, sin operación.

R2 Data Catalog · Iceberg sobre R2

R2 Data Catalog es una capa de tablas Apache Iceberg sobre R2. Metadata catalog administrado por Cloudflare. Consultable desde Snowflake, DuckDB, ClickHouse o Spark vía JDBC/HTTP sin mover datos.

Caso típico en Eigen: eventos de producción (clicks, reservas, transacciones, mensajes de agente) fluyen por Pipelines a Data Catalog. El equipo de análisis consulta con DuckDB desde un notebook. Los reportes mensuales al cliente se generan con SQL estándar.

Analytics Engine · time-series

Analytics Engine es una base time-series integrada para métricas de producto. Escribes data points desde Workers con env.ANALYTICS.writeDataPoint(...), incluyendo dimensiones (customerId, feature, modelo), y consultas con SQL. Escala a millones de points por segundo sin operación.

Este es el reemplazo de Datadog/Grafana para métricas custom de Eigen — latencia por tool-call de agente, conversiones por vertical, uso por cliente. Barato, integrado, consultable.

Logpush · exportación a sistemas externos

Para logs que quieres tener en Datadog, Splunk, S3, GCS, Elasticsearch — Logpush los envía en streaming con formato JSON estructurado. Cada fuente (Workers Trace Events, Firewall, AI Gateway, Zero Trust) puede rutearse a destinos distintos.

RUM · Real User Monitoring

RUM captura Core Web Vitals (LCP, FID, CLS, INP) del tráfico real sin instrumentar JS. Complementado con DEX (Digital Experience Monitoring) para monitoreo sintético desde ubicaciones globales. Este es el stack de performance monitoring para tus landings y frontends.

Bajo el lente de IA

El patrón que Claude Code recomienda por default suele ser "guarda logs en Sentry y métricas en PostHog". Funciona, pero te ata a pricing por usuario activo que escala mal. En Eigen, la doctrina correcta es: errores humanos → Sentry, logs estructurados → Workers Logs + Logpush a R2 (barato, controlable), métricas de producto → Analytics Engine, UX tracking → PostHog. Multiplica por los costos de cada uno y la decisión es obvia. El oficio de Solutions Architect es precisamente este tipo de criterio informado.

§10

Multi-tenant y plataforma SaaS

La diferencia entre un producto y una plataforma son tres primitivas.

El modelo de negocio de Eigen es vertical × multi-tenant — cada cliente (clínica, hotel, farmacia) tiene su propio setup sobre infraestructura compartida. Construir esto correctamente requiere entender tres primitivas de Cloudflare que pocos usan: Workers for Platforms, DO Facets, y Cloudflare for SaaS.

Workers for Platforms · código por cliente

Para casos donde cada cliente necesita lógica distinta — descuentos particulares, reglas de aprobación únicas, formatos de mensaje customizados — no queremos forks por cliente. Workers for Platforms permite hospedar un Worker aislado por tenant dentro de un dispatch namespace común. Un Worker dispatcher resuelve el tenant (por hostname, header, o path) e invoca el Worker correspondiente.

Cada Worker de cliente tiene sus propios límites (CPU, memoria, subrequests), su propio deploy independiente, y aislamiento fuerte contra los demás. Tu plataforma escala a miles de tenants sin tocar el código de ninguno.

DO Facets · SQLite por tenant

DO Facets, la primitiva nueva de abril 2026, permite instanciar Durable Objects con bases SQLite aisladas por entidad dinámica. Cada tenant tiene su propio DO Facet con su propia base — sin pre-provisionamiento, sin schema común forzado. Complementa Workers for Platforms: el código custom escribe a una base custom.

Cloudflare for SaaS · custom hostnames

CF for SaaS maneja el problema clásico de SaaS multi-tenant: cada cliente quiere su propio dominio (acme.com apunta a tu plataforma, no a tu-saas.com/acme). CF emite certificados TLS automáticamente, los rota, y soporta hasta 500,000 custom hostnames por account.

Patrón de plataforma completa

Request → hostname custom (acme.com)
  → Dispatcher Worker (global)
     → resuelve tenantId por hostname
     → invoca Worker aislado del tenant (Workers for Platforms)
        → usa DO Facet con SQLite exclusiva del tenant
        → escribe métricas homogéneas a Analytics Engine central
  → response tipada
Bajo el lente de IA

Workers for Platforms es caro a baja escala. Si tienes 3-5 tenants con lógica similar, un solo Worker con tenantId en los queries es más simple. El salto a Workers for Platforms se justifica a partir de ~20 tenants con divergencia real de lógica. Esta es una decisión arquitectónica — no pidas a Claude que "haga multi-tenant" sin contexto; define el número de tenants y el grado de divergencia esperada.

§11

Observabilidad y operación

Un sistema que no puedes observar no puedes mantener. Punto.

Observabilidad no es logging. Es la capacidad de responder tres preguntas, con evidencia, sobre cualquier incidente: ¿está funcionando bien? ¿cuándo se rompió? ¿por qué? Cloudflare ofrece las primitivas; tu disciplina las convierte en sistema.

Workers Logs y Tail Workers

Workers Logs es logging nativo con retención 3–7 días, búsqueda por correlation ID, filtros. Tail Workers es la primitiva más potente: un Worker que recibe en vivo los eventos de ejecución de otros Workers — logs, traces, errores. Te permite construir pipelines de observabilidad custom sin dependencias externas.

export default {
  async tail(events: TraceEvent[], env: Env) {
    for (const evt of events) {
      if (evt.outcome === "exception") {
        await sendToSentry(evt);
      }
      if (evt.logs.some(l => l.level === "error")) {
        await notifySlack(evt);
      }
      await env.ANALYTICS.writeDataPoint({
        indexes: [evt.scriptName],
        blobs: [evt.event.request?.url ?? ""],
        doubles: [evt.wallTime, evt.cpuTime]
      });
    }
  }
};

Versioned Deploys y Gradual Rollouts

Cada deploy tiene ID inmutable. Puedes hacer rollouts progresivos (10% → 25% → 100%), rollback instantáneo, A/B entre versiones con ruteo determinístico. Esto transforma deploys de "evento ansioso" a "operación rutinaria". La pregunta "¿podemos desplegar al mediodía?" desaparece.

Runbooks y postmortems

La doctrina Eigen: cada servicio productivo tiene un runbook con los 3 escenarios de falla más comunes y cómo responderlos. Cada incidente P0 o P1 termina en un postmortem blameless publicado en /docs/postmortems/. Esto no es tooling — es cultura. Pero las herramientas lo habilitan.

Bajo el lente de IA

Claude Code es excelente escribiendo runbooks a partir de código y postmortems. "Lee este servicio, identifica los 3 failure modes más probables, escribe el runbook en Markdown con pasos de diagnóstico y mitigación." El resultado, con iteración, es material productivo. Pero revísalo — los runbooks incorrectos son peor que no tener runbook, porque le das al on-call una falsa confianza.

§14

Construir con Claude Code sobre Cloudflare

La sección que resume el oficio actual — cómo se trabaja en 2026.

Si llegaste hasta aquí, entiendes la plataforma. Falta la disciplina de construir sobre ella con IA como copiloto. No hay receta mágica. Hay prácticas repetibles.

Configuración del entorno

Cada proyecto de Eigen tiene un CLAUDE.md en la raíz que carga:

  • Una línea sobre qué hace el proyecto.
  • Referencia al manifesto Eigen (doctrina técnica global).
  • Desviaciones del stack default (si aplica).
  • Skills a cargar (ej. cloudflare-workers-patterns, eigen-testing).
  • Comandos críticos (bun dev, bun test, cf deploy).
  • Zonas sensibles que requieren aprobación humana (pagos, migraciones, auth).

Claude Code lee este archivo al inicio de cada sesión. Con MCP oficial de Cloudflare activado, tiene acceso inmediato a ~3,000 operaciones del API — puede crear Workers, desplegar, consultar D1, revisar logs, todo desde conversación.

El loop obligatorio (spec-first)

  1. Brainstorming — entender intent, requirements, trade-offs. No código todavía.
  2. Plan escrito — Markdown con qué, dónde, cómo, tests, rollback. Revisado por humano.
  3. Implementación asistida — con el plan en mano, Claude genera, tú revisas, iteran.
  4. Verificación — tests, smoke manual, correlation IDs en logs, métricas.
  5. Review humano — otro par de ojos, siempre.

Qué aprueba siempre un humano

Arquitectura, invariantes de negocio, cambios de prompt productivo (requieren evals), irreversibles (DB migrations destructivas, cambios de auth), cualquier cosa que toque dinero.

Red flags al aceptar output de IA

SíntomaInterpretación
Tests pasan pero no validan invarianteSilent failure — rechazar
Try/catch que traga erroresTapa un bug — pedir handling explícito
Abstracciones preventivasOver-engineering — simplificar
Cambios fuera del scope del planScope creep — separar o rechazar
Imports o deps no justificadasDependencia fantasma — cuestionar

Prompts productivos: código, no strings

En Eigen los prompts productivos viven en /prompts/*.ts, con template + Zod schema del input. Se versionan con el repo. Se cachean (prompt caching de Anthropic). Se evalúan con suites reales. Un cambio de prompt sin evals actualizadas se rechaza en review como cualquier otro bug.

El cierre

En 2026, la productividad de un ingeniero se mide en intención por minuto: qué tan rápido traduces "necesitamos que X pase" a un sistema funcional. Claude Code multiplica ese factor. Pero la multiplicación solo funciona si tú tienes algo que multiplicar. Criterio arquitectónico, comprensión de trade-offs, disciplina de proceso, capacidad de revisar código con escepticismo productivo — esas son las constantes. Las herramientas cambian cada año; esas habilidades son las que te hacen relevante en 2027, 2028, 2030.
